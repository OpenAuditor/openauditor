# DNS and Domain Security

Your domain name is a critical piece of infrastructure. Attackers who take control of your DNS can redirect all traffic, intercept emails, and generate valid TLS certificates for your domain. This guide covers the controls that protect your domain.

---

## 30-Second Summary

Domain hijacking and DNS spoofing are underappreciated attack vectors. The 2019 Sea Turtle campaign hijacked DNS for government and private sector targets across 40+ organisations. The 2020 incident: attackers hijacked Solana's nameservers and redirected visitors to a phishing site. DNS security is not optional.

---

## Domain Registrar Security

Your first line of defence is securing the account that controls your domain.

### Registrar Account Security

```
Checklist:
✓ Strong, unique password (use a password manager)
✓ Multi-factor authentication — use TOTP (not SMS)
✓ Use a security key (YubiKey) if your registrar supports it
✓ Account recovery email/phone: use a shared team account, not personal
✓ Review authorised users — remove ex-employees
✓ Enable account activity alerts/notifications
```

### Domain Lock

Enable "Registrar Lock" (also called "Domain Lock" or "Transfer Lock"):

```
Effect: Prevents domain transfers without explicit unlock
Where: Registrar control panel → Domain → Lock settings
Verify: WHOIS should show "clientTransferProhibited" status
```

For high-value domains, enable **Registry Lock** (available at some registrars for an extra fee) — requires phone verification to make any changes.

## Email Security (Critical)

Attackers who can send email from your domain can phish your users convincingly. These three DNS records prevent this.

### SPF (Sender Policy Framework)

Specifies which servers are authorised to send email from your domain:

```dns
# DNS TXT record for yourdomain.com
# Format: v=spf1 [include statements] [mechanism] -all

# Example — using Google Workspace
yourdomain.com. TXT "v=spf1 include:_spf.google.com -all"

# Example — using Resend (for transactional email)
yourdomain.com. TXT "v=spf1 include:spf.resend.com -all"

# Multiple providers
yourdomain.com. TXT "v=spf1 include:_spf.google.com include:spf.resend.com -all"

# -all = reject (preferred) — fail mail from any server not listed
# ~all = softfail (weaker) — mark as suspicious but don't reject
# +all = pass all (never use) — useless
```

Verify: `dig TXT yourdomain.com` or `nslookup -type=TXT yourdomain.com`

### DKIM (DomainKeys Identified Mail)

Your email provider signs outgoing emails with a private key. Recipients verify with the public key in your DNS:

```dns
# Generated by your email provider (Google, SendGrid, Resend, etc.)
# Example format — get the actual key from your email provider's dashboard:
google._domainkey.yourdomain.com. TXT "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3..."

# Multiple providers need multiple DKIM records with different selectors
# Google uses "google._domainkey"
# SendGrid uses "s1._domainkey" and "s2._domainkey"
```

Setup: Follow your email provider's DKIM setup guide — they generate the key pair.

### DMARC (Domain-based Message Authentication)

Tells receiving mail servers what to do when SPF and DKIM checks fail:

```dns
# Start with p=none (monitoring mode) — collect reports, don't block yet
_dmarc.yourdomain.com. TXT "v=DMARC1; p=none; rua=mailto:dmarc-reports@yourdomain.com"

# After reviewing reports with no false positives, move to quarantine:
_dmarc.yourdomain.com. TXT "v=DMARC1; p=quarantine; pct=100; rua=mailto:dmarc@yourdomain.com"

# Final target — reject all failing emails:
_dmarc.yourdomain.com. TXT "v=DMARC1; p=reject; pct=100; rua=mailto:dmarc@yourdomain.com; ruf=mailto:dmarc-forensic@yourdomain.com"

# p=reject is the gold standard — prevents phishing using your domain name
```

**Rollout plan:**
1. Week 1: `p=none` — collect DMARC reports
2. Week 2: Review reports at `dmarc-reports@yourdomain.com` or use a service like Postmark DMARC or MXToolbox
3. Week 3: `p=quarantine; pct=10` — quarantine 10% of failing mail
4. Week 4: `p=quarantine; pct=100`
5. Week 6: `p=reject` — full enforcement

### BIMI (Optional — Brand Indicator)

Shows your logo in email clients for verified senders. Requires DMARC at `p=reject` and a VMC certificate:

```dns
default._bimi.yourdomain.com. TXT "v=BIMI1; l=https://yourdomain.com/logo.svg; a=https://yourdomain.com/vmc.pem"
```

## DNSSEC

DNSSEC digitally signs DNS records to prevent spoofing. Attackers can't forge DNS responses for DNSSEC-protected domains.

```bash
# Check if DNSSEC is enabled for your domain
dig +dnssec yourdomain.com

# Look for:
# - RRSIG records (signatures) in the response
# - ad flag in the response header (authenticated data)

# Enable in registrar settings: DNS → DNSSEC
# Then your registrar and DNS host coordinate key exchange
```

Not all registrars support DNSSEC. Cloudflare supports it for free.

## Cloudflare as a Security Layer

Cloudflare (free tier) adds several security benefits:

1. **DDoS protection** — absorbs attack traffic before it reaches your servers
2. **DNS proxying** — hides your origin server IP
3. **Rate limiting** — configurable at the network level
4. **Bot management** — blocks automated attacks

```
Setup:
1. Create Cloudflare account
2. Add your domain
3. Update nameservers at your registrar to Cloudflare's
4. Configure SSL/TLS mode: Full (strict)
5. Enable "Always use HTTPS" redirect
6. Enable Brotli compression
7. Set minimum TLS version to 1.2
```

### Key Cloudflare settings

```
SSL/TLS:
  Mode: Full (strict)         # Never "Flexible" — it's worse than no Cloudflare for auth
  Min TLS version: TLS 1.2
  HSTS: enable (max-age=31536000, includeSubDomains, preload)
  
Security:
  Security level: Medium      # Challenges known bad actors
  Bot Fight Mode: On          # Free bot protection

Firewall Rules (examples):
  - Block countries you don't operate in
  - Rate limit login endpoint: 10 requests per minute
  - Block requests to /wp-admin, /.env, /etc/passwd
```

## Subdomain Security

### Subdomain Takeover

If you create a subdomain that points to an external service (Heroku, GitHub Pages, Netlify) and then delete the service without removing the DNS record, attackers can claim that service and serve content from your subdomain.

```bash
# Check for dangling DNS records
# Install subjack or subzy
go install github.com/haccer/subjack@latest
subjack -w subdomains.txt -t 100 -timeout 30 -ssl -c fingerprints.json

# Or use: https://github.com/EdOverflow/can-i-take-over-xyz
# Lists which services are vulnerable to subdomain takeover
```

**Prevention:**
- Remove DNS records before decommissioning services
- Audit subdomains quarterly: `dig NS subdomain.yourdomain.com`
- Require approval to create new DNS records

### Wildcard Certificates and DNS

```
Don't use wildcard DNS (*.yourdomain.com → same IP)
unless every subdomain should be served by the same app.

Risk: A subdomain takeover on any subdomain is trivially achievable
if the wildcard resolves but the service doesn't exist.
```

## Audit Checklist

- [ ] Domain lock enabled at registrar
- [ ] Registrar account has MFA (TOTP, not SMS)
- [ ] SPF record published and validated
- [ ] DKIM configured for all email providers
- [ ] DMARC at `p=reject` (or at minimum `p=quarantine`)
- [ ] DNSSEC enabled (if registrar supports it)
- [ ] HTTPS enforced (301 redirect from HTTP)
- [ ] HSTS header with at least 1-year max-age
- [ ] No dangling DNS records for decommissioned services
- [ ] Wildcard DNS records reviewed and justified
- [ ] DNS changes require approval and are logged
